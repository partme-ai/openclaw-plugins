# 微信客服（WeCom KF）原理与约束（面向智能客服/转人工）

本文档基于企微官方文档整理微信客服的消息机制、会话状态机与关键约束，作为 openclaw-wecom-ics 的实现口径。

## 1. 总体机制：回调通知 + 主动拉取消息

微信客服的核心是“事件回调只做通知，消息内容需主动拉取”：

- 当微信客户、接待人员发消息或有行为动作时，企微后台会向企业配置的回调 URL 推送事件（如 `kf_msg_or_event`）。
- 业务侧收到回调后，使用回调中携带的信息（尤其是 `open_kfid` 与 `token`）调用 `kf/sync_msg` 主动读取最近 3 天内的具体消息与事件。

关键点：

- 回调事件中的 `token` 仅用于拉取接口的合法性校验，**10 分钟内有效**；若不填 token，`sync_msg` 将受到更严格的频率限制。
- `sync_msg` 使用 `cursor` 增量拉取：首次不填 cursor，会从 3 天内最早的消息开始返回；后续用 `next_cursor` 继续拉取，直到 `has_more=0`。

参考：

- 接收消息和事件：<https://developer.work.weixin.qq.com/document/path/94670>

## 2. 消息读取：kf/sync_msg（94670）

### 2.1 能读到什么

可读取最近 3 天内：

- 客户上行消息
- 接待人员回复消息
- 发送消息接口的“发送失败事件”（例如用户拒收）
- 客户点击菜单消息的回复消息

不支持读取：

- 通过“发送消息接口”发送出去的消息本身（只能读到失败事件等）

支持类型（以官方为准）：文本、图片、语音、视频、文件、位置、链接、名片、小程序、菜单、事件等。

### 2.2 关键约束

- `limit`：默认/最大均为 1000；返回条数可能小于 limit，需要结合 `has_more` 判断是否继续。
- `voice_format`：0=Amr，1=Silk。
- 媒体大小限制（超限会返回文本提示消息）：
  - 图片：2MB
  - 语音：2MB
  - 视频：10MB
  - 文件：20MB
- 建议实现“去重 + 游标持久化”：
  - 以 `msgid` 做幂等去重（避免回调重试导致重复处理）
  - 以 `open_kfid` 维度持久化 `cursor`（不同客服账号独立推进）

参考：

- 接收消息和事件：<https://developer.work.weixin.qq.com/document/path/94670>

## 3. 发送消息：kf/send_msg（94677）

### 3.1 什么时候能发

当客户处于以下会话状态时，允许通过 API 下发消息：

- “新接入待处理”
- “由智能助手接待”

并且存在双重约束：

- **48 小时窗口**：仅当客户在主动发送消息给客服后的 48 小时内可下发。
- **条数上限**：在上述 48 小时内最多可下发 **5 条**；若用户继续发送消息，可再次下发。

### 3.2 成功返回不等于送达

`send_msg` 返回 `errcode=0` 仅代表接口调用成功，不代表消息最终送达；需要关注“消息发送失败事件”的回调与 `sync_msg` 拉取结果做兜底判定。

参考：

- 发送消息：<https://developer.work.weixin.qq.com/document/path/94677>

## 4. 事件响应消息：kf/send_msg_on_event（95122）

该接口用于发送“欢迎语/排队提示/非工作时间提示/结束语/满意度”等**事件场景**消息，特点是：

- 以 `code` 为凭证（从事件回调或会话状态变更接口获得）
- `code` **仅可使用一次**
- `code` 有严格有效期与会话状态约束：必须在获取 code 对应的会话状态下发送

### 4.1 常见场景与有效期

官方约束（以文档为准）：

- 用户进入会话（欢迎语）：1 条，**20 秒**，文本/菜单；事件回调获得 `welcome_code`
- 进入接待池（排队提示等）：1 条，**48 小时**，文本；通过会话转接/状态变更获得 code
- 从接待池接入会话（非工作时间/超时未回复提示等）：1 条，**48 小时**，文本；事件回调或状态变更获得 code
- 结束会话（结束语/满意度）：1 条，**20 秒**，文本/菜单；事件回调或状态变更获得 code

参考：

- 发送欢迎语等事件响应消息：<https://developer.work.weixin.qq.com/document/path/95122>

## 5. 会话状态机与转人工：kf/service_state/*（94669）

### 5.1 状态定义

一次会话从用户发起咨询到结束，会在以下状态间流转（ID 以官方为准）：

- 0：未处理（新会话接入，可自动回复/入待接入池/指定接待人员）
- 1：由智能助手接待（可 API 回复，可转入待接入池或指定接待人员）
- 2：待接入池排队中
- 3：由人工接待
- 4：已结束/未开始会话

### 5.2 关键约束与接口关系

- 获取会话状态：`kf/service_state/get`
- 变更会话状态：`kf/service_state/trans`
  - 当 `service_state=3`（指定人工坐席）时，`servicer_userid` 必填且该坐席必须处于“正在接待”。
  - 会话初次变更到 `2/3/4` 时，会返回 `msg_code`，用于调用 `send_msg_on_event` 发送排队提示、非工作时间提示、结束语等事件响应消息。
  - 若接待人员未在企微客户端激活使用，可能返回 `95014` 错误（转人工前需做可用性校验与降级）。

参考：

- 分配客服会话：<https://developer.work.weixin.qq.com/document/path/94669>

## 6. 客服账号与坐席：account/list、add_contact_way、servicer/list

### 6.1 获取客服账号列表：kf/account/list（94661）

- 分页参数：`offset`（默认 0）、`limit`（默认 100，范围 1~100）
- 返回 `open_kfid`、名称、头像等；部分场景会带 `manage_privilege`

参考：

- 获取客服账号列表：<https://developer.work.weixin.qq.com/document/path/94661>

### 6.2 获取客服账号链接：kf/add_contact_way（94665）

- 可指定 `scene`，用于区分来源；返回链接可拼接 `scene_param=SCENE_PARAM`
- **严禁**修改或复制参数到其他链接使用，否则“用户进入会话事件”参数校验不通过，导致无法回调

参考：

- 获取客服账号链接：<https://developer.work.weixin.qq.com/document/path/94665>

### 6.3 获取接待人员列表：kf/servicer/list（94645）

- 以 `open_kfid` 查询 `servicer_list[]`
- `status`：0=接待中，1=停止接待（可能包含暂时挂起等子类型）

参考：

- 获取接待人员列表：<https://developer.work.weixin.qq.com/document/path/94645>

## 7. 回调通知：客服账号授权变更（97712）

当应用授权的客服账号发生变化（新增/取消授权）会触发回调：

- `Event = kf_account_auth_change`
- `AuthAddOpenKfId`：新增授权列表
- `AuthDelOpenKfId`：取消授权列表

该事件用于：

- 维护“当前可通过 API 管理的 open_kfid 集合”
- 触发游标初始化/清理（按 open_kfid 维度）

参考：

- 回调通知：<https://developer.work.weixin.qq.com/document/path/97712>

## 8. openclaw-wecom-ics 落地约束（实现口径）

为保证一致性与可运维性，插件侧建议遵循以下约束：

1. **回调必须快速 ACK**：回调仅做通知入队，不在回调请求链路中执行 LLM 推理与多次拉取。
2. **拉取按 open_kfid 串行推进**：同一 `open_kfid` 的 `cursor` 只能单向前进，避免并发拉取导致游标回退或重复消费。
3. **幂等处理**：以 `msgid` 作为去重键，避免回调重试/重复拉取导致重复回复、重复转人工。
4. **发送能力选择**：
   - 普通回复：优先使用 `send_msg`，并严格遵守 48 小时/5 条限制与会话状态限制。
   - 欢迎语/排队/结束语等固定提示：使用 `send_msg_on_event`，并在 code 有效期内一次性发送。
5. **转人工策略**：
   - 转入排队：`service_state=2`（可配合 `msg_code` 发送排队提示）
   - 指定坐席：`service_state=3`，坐席需来自 `servicer/list` 且处于接待中；失败时降级为排队或继续机器人接待
6. **账号治理**：以 `kf_account_auth_change` 回调为准维护授权账号列表；对未授权账号拒绝拉取/发送，避免产生 403/权限类错误风暴。
