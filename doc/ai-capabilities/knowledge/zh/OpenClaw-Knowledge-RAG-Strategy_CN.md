## 1. 增值定位：纯加法，不侵入

本插件的底层逻辑：**`@partme.ai/openclaw-knowledge` 是一个独立的知识库 RAG 引擎，不依赖任何特定渠道插件，通过标准 API 集成到任意渠道的消息流中：**

```text
消息入站 → 渠道插件处理 → [before_prompt_build 钩子] → 回复出站
                                    ↑
                               注入 RAG 上下文
                               + 知识库 Tool 注册
```

**什么叫"纯加法"：**

- 不修改渠道插件的核心文件
- 所有知识库功能通过 OpenClaw 的 `before_prompt_build` hook 注入 RAG 上下文
- 知识库 CRUD 通过 `api.registerTool()` 注册 4 个标准 Tool：`knowledge_add`、`knowledge_query`、`knowledge_update`、`knowledge_delete`
- 独立启用/禁用，不破坏原有的 Bot 优先 / Agent 回退逻辑

**为什么能作为商业产品卖：**

- **通用性强**——知识库、业务对接、群聊整理，几乎每家企业都有这个需求
- **独立可部署**——知识库 RAG 不依赖任何第三方系统，零依赖启动
- **多渠道兼容**——一个知识库引擎，适配企业微信、飞书、钉钉、QQ 机器人、微信公众号等主流 IM 渠道
- **可扩展**——企业私有业务通过标准 MCP 协议接入，信创适配即可
- **配置层零代码**——管理员只需配置几个 JSON 字段

### 纯加法的 5 个合法扩展点

所有知识库功能只能通过以下 5 个入口注入，**绝不侵入核心文件**：

| 扩展点 | 技术入口 | 商业用途 | 状态 |
|--------|---------|----------|------|
| **1. 知识库上下文注入** | `before_prompt_build` hook | 在 AI 处理消息前注入 RAG 检索结果和自定义指令 | ✅ 已实现 |
| **2. 工具注册入口** | `api.registerTool()` | 注册 `knowledge_*` 工具，让 AI 能自主读写知识库 | ✅ 已实现 |
| **3. 消息预处理器** | 增值前处理拦截点（消息解析之后、AI 处理之前） | 消息过滤、审计日志、流量控制、敏感信息检查等 | 🔲 预留 |
| **4. 渠道 API 客户端** | 渠道平台 API 封装层 | AccessToken 管理、媒体上传等基础能力，供 MCP 层和自定义业务逻辑调用 | ✅ 已有 |
| **5. 技能目录** | 技能（Skills）目录 | 编写 MCP 对接说明书（SKILL.md），指导 AI 正确使用业务工具 | ✅ 已有 |

**核心文件保护原则**——以下逻辑绝不修改：

| 文件/模块 | 原则 | 对客户的意义 |
|-----------|------|-------------|
| 消息流核心 | ❌ 不修改 | 消息处理不受知识库功能影响 |
| 渠道生命周期 | ❌ 不改 | 渠道稳定性不受知识库功能影响 |
| MCP 传输层 | ❌ 不改 | 传输层已完备，新增业务品类无需修改传输代码 |

**为什么这对客户重要**：纯加法意味着知识库插件可以独立购买、独立安装、独立卸载。今天用企微渠道加知识库，明天切换到飞书渠道——知识库能力不变，只需改配置路径。

## 2. 三层能力模型

> **核心洞察**：`@partme.ai/openclaw-knowledge` 的能力体系是**三层递进**的——从"知道"到"多渠道覆盖"再到"懂业务"。每一层都独立可交付。

| 层次 | 能力名称 | 核心价值 | 适用渠道 | 数据源 |
|------|----------|----------|----------|--------|
| **L1** | 知识库 RAG | AI 有脑子了 | 所有渠道 | 用户上传文档 → 切片 → Embedding → 向量库 |
| **L2** | 多渠道集成适配 | 一套引擎，多端部署 | 企微/飞书/钉钉/QQBot/微信 | 各渠道文件上传、文档 API |
| **L3** | 企业私有业务接入 | AI 懂你的业务了 | 所有渠道 | 企业自有系统（HR/工单/审批/数据库） |

### 协同关系图

```text
                        ┌─────────────────────────────────────────┐
                        │            before_prompt_build            │
                        │           (单次 Agent 请求)               │
                        └──────┬─────────────┬────────────────────┘
                               │             │
                    ┌──────────▼──┐    ┌─────▼──────────┐
                    │  RAG 检索    │    │  知识库 Tool    │
                    │ (知识库向量)  │    │ (add/query/    │
                    └──────┬──────┘    │  update/delete) │
                           │           └─────┬──────────┘
                    ┌──────▼──────────────────▼──────┐
                    │        Agent 模型推理           │
                    │  （system prompt = RAG 上下文    │
                    │   + tools = knowledge_* 工具）   │
                    └───────────────────────────────┘
                               │
                    ┌──────────▼──────────┐
                    │   Agent 回复输出      │
                    │（含溯源引用 + 操作确认）│
                    └─────────────────────┘
```

## 3. 第一层：知识库 RAG（"AI 有脑子了"）

### 为什么通用

每家公司都有自己的文档、制度、产品说明。不需要对接任何第三方系统，只需要管理员上传文档或指定文件目录。

### 用户视角

```text
管理员：在 knowledge 配置中启用知识库，把制度文档放进指定目录
Bot：    用户问问题 → 自动搜文档 → 结合上下文回复
管理员：不用了，关闭 enabled 即可
```

### 商业包装

- **零配置知识库**：上传文件即用，不用额外建站、不用写爬虫
- **自动索引**：上传文件自动切片、Embedding、入库
- **无缝集成**：用户无感知，和普通聊天一样问问题，AI 自动判断要不要查知识库

### 具体功能点

#### 数据来源：两条数据通路

知识库 RAG 的数据来源分为两条相互隔离的通路：

| 通路 | 数据源 | 触发方式 | 索引目标 | 使用场景 |
|------|--------|----------|----------|----------|
| **通路 A** | 用户上传文件（对话附件） | 用户发送文件 → Bot/Agent 自动索引 | `accountId:mode` 对话级私有 namespace | 用户与 Bot 共享上下文，不沉淀为全局知识 |
| **通路 B** | 管理员指定文档目录 | 手动索引命令或定时调度 | `accountId:enterprise` 企业级全局 namespace | 公司制度、产品文档等需要全员可查的知识 |

> **关键设计约束**：通路 A 和通路 B 写入不同的 namespace，数据不混淆。

| 功能 | 描述 | 实现思路 |
|------|------|----------|
| **文档自动索引（通路 B）** | 管理员指定文档目录/文件 ID，定期索引到全局知识库 | 定时任务 + 切片器 |
| **对话上下文增强** | @机器人时，检索相关知识注入 system prompt | `before_prompt_build` hook |
| **智能回复溯源** | 回复末尾带引用来源（"参考了《XX制度》第3章"） | 检索结果附带元信息（sourceId + chunkIndex） |
| **智能摘要 + 提炼** | 长对话自动提炼要点，存入向量库供后续检索 | 定时任务触发 LLM 摘要 → Embedding → upsert |
| **知识库搜索建议** | 用户输入时，自动建议相关知识库文档标题 | 标题模糊匹配 + 向量检索 |
| **多租户隔离** | 按 `accountId:mode` 命名空间隔离 | `namespace` 生成逻辑 + 数据来源路由 |

### 技术架构

```text
┌─────────────────────────────────────────────────┐
│               before_prompt_build hook           │
│  ┌──────────┐    ┌──────────┐    ┌────────────┐  │
│  │ Chunker  │ →  │Embedding │ →  │  Retriever │  │
│  │(文档切片) │    │(向量化)   │    │  (检索)     │  │
│  └──────────┘    └──────────┘    └────────────┘  │
│       ↑                ↓                ↓        │
│  ┌──────────┐    ┌──────────┐    ┌────────────┐  │
│  │ 渠道文档 │    │ ZVec /   │    │  System     │  │
│  │ (文件上传)│    │ SQLite   │    │  Prompt     │  │
│  │          │    │ (向量库)  │    │  (上下文)    │  │
│  └──────────┘    └──────────┘    └────────────┘  │
└─────────────────────────────────────────────────┘
```

**已有实现**（约 1800 行 TS）：

- `src/knowledge/embedding/openai.ts` — Embedding 接口（OpenAI 兼容）
- `src/knowledge/store/zvec.ts` — 纯 JS 向量存储（零依赖，开箱即用）
- `src/knowledge/store/sqlite-vec.ts` — SQLite-Vec 后端（适合生产）
- `src/knowledge/indexer/chunker.ts` — 文档切片（按段落/标题/Token）
- `src/knowledge/retriever/hybrid.ts` — 混合检索（向量 + 关键词）
- `src/knowledge/hooks.ts` — `before_prompt_build` hook 注入点

## 4. 第二层：多渠道集成适配

> 本层定义了 `@partme.ai/openclaw-knowledge` 的**渠道无关性**设计——同一套知识库引擎，通过标准集成接口适配不同 IM 渠道。

### 渠道无关设计原则

| 原则 | 说明 |
|------|------|
| **配置路径通用化** | 知识库配置统一位于 `channels.{channel}.knowledge.*`，渠道名称由集成方指定 |
| **工具命名统一** | 所有知识库工具统一命名为 `knowledge_*`（非 `wecom_knowledge_*`），跨渠道复用 |
| **Hook 通道过滤** | `before_prompt_build` hook 内部按 `ctx.channelId` 自动过滤，不同渠道互不干扰 |
| **按需注册工具** | 各渠道可选择性注册全部 4 个工具或部分工具，按业务场景裁剪 |

### 支持的渠道

| 渠道 | 配置路径 | 推荐注册工具 | 说明 |
|------|----------|-------------|------|
| **企业微信（wecom）** | `channels.wecom.knowledge` | add + query + update + delete | 完整 CRUD，适合企微办公场景 |
| **飞书（Lark）** | `channels.lark.knowledge` | add + query | 基础写入和检索，适合文档协作 |
| **钉钉（DingTalk）** | `channels.dingtalk.knowledge` | add + query + update | 文本写入、检索和更新 |
| **QQ 机器人（QQBot）** | `channels.qqbot.knowledge` | add + query | 快速问答场景 |
| **微信公众号（Weixin）** | `channels.weixin.knowledge` | query（仅检索） | 被动回复模式，仅需自动检索 |

### 集成代码模板

```typescript
import { registerKnowledgeHooks, createKnowledgeAddTool, createKnowledgeQueryTool,
  createKnowledgeUpdateTool, createKnowledgeDeleteTool } from '@partme.ai/openclaw-knowledge';

export function onRegister(api: PluginApi) {
  registerKnowledgeHooks(api, 'channels.{channel}.knowledge');
  api.registerTool(createKnowledgeAddTool);
  api.registerTool(createKnowledgeQueryTool);
  api.registerTool(createKnowledgeUpdateTool);
  api.registerTool(createKnowledgeDeleteTool);
}
```

### 渠道适配的差异性

不同渠道在知识库使用上存在差异，插件需适配以下方面：

| 维度 | 企微 | 飞书 | 钉钉 | QQBot | 微信 |
|------|------|------|------|-------|------|
| 文件上传 | 支持多格式 | 支持多格式 | 支持 | 支持图片 | 仅文本 |
| 主动推送 | ✅ | ✅ | ✅ | ✅ | ❌（被动回复） |
| 文档 API | 企微文档库 | 飞书文档 | 钉盘 | 无 | 无 |
| 最大文件 | 25MB | 50MB | 20MB | 20MB | 10MB |

## 5. 第三层：企业私有业务接入（"AI 懂你的业务了"）

在第一层（知识库 RAG）和第二层（多渠道集成）的基础上，第三层解决的是**企业特有的业务系统对接**问题。不是做成通用插件，而是提供**接入框架 + 标准接口**，让企业或集成商可以低成本对接自己的系统。

### 架构模式

```
┌──────────────────────────────────────────────────────────────┐
│                    @partme.ai/openclaw-knowledge               │
│  ┌──────────────┐  ┌──────────────────┐  ┌────────────────┐  │
│  │ L1: 知识库   │  │ L2: 多渠道适配   │  │ L3: 企业私有    │  │
│  │     RAG      │  │    企微/飞书/钉钉 │  │    HR/工单/审批 │  │
│  │              │  │    /QQBot/微信    │  │    /SQL 查询    │  │
│  └──────────────┘  └──────────────────┘  └────────────────┘  │
│                           │                    │              │
│                           ▼                    ▼              │
│                    ┌──────────────┐  ┌──────────────────┐     │
│                    │  渠道消息流   │  │  企业私有 MCP     │     │
│                    │ (各 IM 平台)  │  │ (自定义品类)      │     │
│                    └──────────────┘  └──────────────────┘     │
└──────────────────────────────────────────────────────────────┘
```

### 具体接入场景

| 场景 | 描述 | 实现方式 | MCP 工具示例 |
|------|------|----------|-------------|
| **AI 工单系统对接** | 员工@机器人创建工单、查状态、催单 | 注册自定义 `ticket` MCP 品类，对接企业工单系统 API | `ticket.create`, `ticket.getStatus`, `ticket.escalate` |
| **数据查询 MCP（SQL→自然语言）** | 自动写 SQL 查数据库，用自然语言返回结果 | 注册自定义 `sql` MCP 品类，后端接数据库（需 SQL 生成 + 安全审计） | `sql.query`, `sql.explain`, `sql.schema` |
| **审批流集成** | 查审批进度、审批链、待办审批 | 注册自定义 `approval` MCP 品类，对接 OA/审批系统 | `approval.list`, `approval.getDetail`, `approval.chain` |
| **HR 系统对接** | 查假期余额、工资条、组织架构 | 注册自定义 `hr` MCP 品类，对接 HR 系统 API 或数据库 | `hr.leaveBalance`, `hr.payroll`, `hr.orgChart` |
| **CRM 客户查询** | 查客户信息、合同状态、商机进展 | 注册自定义 `crm` MCP 品类，对接 CRM 系统 | `crm.getCustomer`, `crm.listOpportunities` |

### 接入原则

1. **MCP 协议统一**：所有私有系统统一注册为标准 MCP 品类，复用已有的传输层和 session 管理
2. **安全第一**：私有 MCP 必须做鉴权、审计和 SQL 注入防护
3. **Skill 驱动**：每个私有品类对应一个 Skill 目录，方便 Agent 选择合适的工具
4. **渐进式接入**：从最简单的"查状态"开始，逐步扩展到"创建/修改"操作

## 6. 三个可落地原型

### 原型 A：HR 助手

```text
用户：@机器人 "我年假还剩几天"
内部流程：
  1. before_prompt_build hook → 检索知识库（员工手册 PDF 中关于年假的规定）
  2. Agent 调用 knowledge_query 检索相关制度
  3. 组合 RAG 上下文 → LLM 推理 → 回复
Bot 回复："根据《员工手册》第三章，年假按工龄计算。
           您目前剩余年假 3.5 天（总 5 天，已休 1.5 天）。"
```

### 原型 B：群聊知识助手

```text
用户：@机器人 "上次那个微服务降级方案谁有链接？"
内部流程：
  1. 检索群聊历史向量库（最近 N 条已向量化消息）
  2. 检索知识库（文档索引）
  3. 组合上下文 → LLM 推理
Bot 回复："找到了！该方案由 @张三 在上周二（3月15日）分享：
           方案摘要：基于 Sentinel 的熔断降级，核心策略包括...（以下省略）"
```

### 原型 C：数据查询助手

```text
用户：@机器人 "这个月工单完成率"
内部流程：
  1. Agent 调用知识库检索相关统计模板
  2. 组合上下文 → LLM 推理
Bot 回复："📊 本月工单统计（截至4月24日）：
           - 总工单：156 个
           - 已完成：132 个（完成率 84.6%）
           - 进行中：18 个
           - 逾期：6 个（逾期率 3.8%）
           
           较上月完成率（78.2%）提升 6.4 个百分点。"
```

| **已就绪** | 文档切片、向量存储、混合检索、AI 自动回答 | 核心引擎渠道无关，可集成至任意渠道 | 开箱即用，今天部署今天见效 |
| **v0.1** | 文档自动同步：配置指定目录，自动索引 | 企微 + 飞书集成示例可用 | 一套知识库，两个渠道 |
| **v0.2** | 对话上下文增强：AI 记得对话历史 | 钉钉 + QQBot 集成 | 对话记忆 + 更多渠道 |
| **v0.3** | 长对话智能摘要：AI 自动提炼要点 | 微信适配 | 渠道全覆盖 |

**每侧都可独立升级**：客户可以只升级知识库到 v0.2 而不升级多渠道侧，反之亦然。

## 8. 商业包装要点

### 定价模型建议

| 层 | 内容 | 建议定价 | 说明 |
|----|------|----------|------|
| 基础 | 知识库引擎核心（开源基础版） | 免费 | 社区版，MIT 许可证 |
| 知识库 | RAG 知识库（按文档量/企业数） | 按席位或按文档容量 | 含向量存储、切片、检索 |
| 多渠道 | 多渠道集成适配包 | 按渠道数或捆绑 | 企微/飞书/钉钉/QQBot/微信 |
| 企业私有 | 自定义 MCP 品类接入框架 | 按品类数或按调用量 | 需企业自行开发后端 MCP Server |

### 交付模式

- **插件包**：`npm install @partme.ai/openclaw-knowledge`，作为依赖引入渠道插件
- **配置驱动**：所有知识库功能通过 `channels.{channel}.knowledge.*` 配置项控制
- **许可证**：商业许可证（非开源），可搭配试用期

### 差异化卖点

| vs 竞品 | 我们的优势 |
|---------|-----------|
| vs 通用 RAG 方案 | 深度集成 OpenClaw 生态，原生支持 `before_prompt_build` hook 和 Tool 注册 |
| vs 平台绑定方案 | 渠道无关，一套引擎适配所有 IM 平台，迁移成本为零 |
| vs 自建方案 | 开箱即用，不需要开发团队；零配置起步，渐进式增强 |

### 常见销售场景

1. **HR 场景**："你的机器人能查年假吗？" → 原型 A + 知识库 RAG
2. **知识管理场景**："新员工培训，机器人能回答公司制度问题吗？" → 知识库 RAG
3. **跨平台场景**："我们同时用企微和飞书，能不能用同一套知识库？" → 多渠道集成
4. **效率提升场景**："群里的日报能不能自动汇总？" → 智能摘要 + 知识库
5. **业务赋能场景**："机器人能帮我们查工单、催审批吗？" → 私有业务 MCP
